Direct Discovery Endpoint Bypasses ResourceLock Signature Validation

The solver has two ways an off-chain order can enter the system. The safe public API path validates ResourceLock EIP-712 signatures before forwarding. The discovery service also exposes its own POST /intent endpoint, but that direct path queues the intent without running the same signature validator.

Expected gate Bypass path Impact: invalid intent reaches solver engine
Public safe entry /api/v1/orders
Direct risky entry /intent
Missing check EIP-712 signature

How the Two HTTP Servers Work

There are two Axum servers involved. The first one is the main solver-service API, which is the server operators usually expose to clients. The second one is the off-chain discovery server, which exists so the solver engine can receive intents through the discovery subsystem.

Main solver-service API

Typical role
Public-facing API.
Routes
/api/v1/orders, /api/v1/quotes, /api/v1/admin/*, /health
Important behavior
For ResourceLock orders, it runs signature validation before forwarding.

Off-chain discovery API

Typical role
Internal ingestion server for discovered off-chain intents.
Routes
/intent only.
Important behavior
It converts the request into an intent and queues it, but does not run the ResourceLock signature validator.

The safe flow is public client → solver-service → discovery. The risky flow is attacker → discovery directly. If discovery is truly private, this is mostly an internal trust-boundary bug. If discovery is bound to 0.0.0.0 or exposed by Docker, Kubernetes, a load balancer, or firewall rules, it becomes externally reachable.

Safe Path: Public API

Step 1

Client submits order to solver-service

The normal external path is POST /api/v1/orders.

Step 2

ResourceLock signature is verified

SignatureValidationService checks the EIP-712 signature against the order user before the order is forwarded.

Step 3

Validated request is forwarded

Only after validation does solver-service forward the request to the off-chain discovery URL.

Result

Forged ResourceLock orders are rejected early

A bad signature fails before the order reaches the solver engine.

Bypass Path: Direct Discovery

Step 1

Attacker reaches discovery service directly

If the discovery API is bound to a reachable interface, the attacker can send to POST /intent instead of /api/v1/orders.

Step 2

Handler parses and extracts sponsor

For ResourceLock orders, sponsor extraction trusts the order payload's sponsor field. This is not the same as proving the sponsor signed the order.

Step 3

Signature validation is skipped

The direct handler computes the order ID and sends the intent to the internal queue without calling SignatureValidationService.

Result

Invalid intent reaches solver processing

The engine may spend RPC, CPU, queue capacity, simulation effort, and possibly transaction attempts before the invalid signature causes on-chain failure.

Why this matters

The bug is not that the order has no signature field. The bug is that one entry point proves the signature is valid, while the other only carries signature bytes forward. Carrying bytes is not verification.

Attacker Shape

An attacker does not need to break cryptography. They only need network access to the discovery API.

POST http://discovery-host:8081/intent
Content-Type: application/json

{
  "order": {
    "type": "oif-resource-lock-v0",
    "payload": {
      "message": {
        "sponsor": "0xVictimOrFakeSponsor",
        "outputs": "...valid-shaped route data..."
      }
    }
  },
  "signature": "0xInvalidOrUnrelatedSignature",
  "quoteId": null
}

What the Code Does

  • /api/v1/orders calls signature validation for ResourceLock orders before forwarding.
  • /intent converts the request into an intent and sends it to the solver queue.
  • Downstream order validation checks shape, expiry, supported routes, and order ID, but not the EIP-712 signature validity.
  • For Compact finalization, the signature is later included in transaction calldata. Invalid signatures can then fail late, after solver resources were consumed.

Practical Impact

  • Unauthorized or invalid orders can enter the solver's internal processing pipeline.
  • Attack traffic can consume RPC quota because order ID computation calls settler contracts.
  • The unbounded channel can accumulate work faster than the solver drains it.
  • The solver may simulate, price, or attempt transactions for orders that should have been rejected at the edge.

Fix Direction

  • Best fix: run the same ResourceLock signature validation inside the discovery /intent handler.
  • Defense in depth: require internal auth, mTLS, or bind discovery to localhost/private networking only.
  • Operational hardening: add rate limits and replace the unbounded channel with bounded backpressure.
  • Regression test: direct /intent rejects a ResourceLock order with an invalid signature.

How Costly Is the Fix?

The minimum security fix is small. The broader hardening work is still manageable, but it touches more operational assumptions.

Low cost: validate in discovery

Reuse the existing ResourceLock signature validator in the /intent handler and add a regression test for invalid signatures. This is the best direct fix.

Low cost: restrict exposure

Bind discovery to 127.0.0.1 or private networking and make sure deployment manifests do not publish its port. This reduces reachability risk but does not fix the code path.

Medium cost: full hardening

Add internal auth or mTLS, rate limits, and bounded queue backpressure. This addresses both bypass and resource-abuse risk.

Recommended Documentation Wording

The intended architecture should be documented explicitly so operators understand the trust boundary: 

The off-chain discovery API is an internal ingestion endpoint. It must not be
exposed to untrusted networks. Public clients should submit orders through the
main solver-service API at /api/v1/orders, which performs ResourceLock
signature validation before forwarding to discovery.

Deployments must bind the discovery API to localhost/private networking or
protect it with internal authentication. As defense in depth, the discovery
/intent handler should still reject invalid ResourceLock signatures so a future
network misconfiguration does not create an external bypass.

With that threat model, the issue is best described as an internal trust-boundary hardening bug unless the discovery port is exposed. If exposed, it becomes externally exploitable.
