Public Registration Can Mint Admin-Read JWTs

The registration endpoint is meant to let public clients self-register for normal API scopes. The bug is that the public scope parser rejects admin-all but still accepts admin-read. That creates a read-only admin privilege escalation when public registration and admin routes are enabled.

Root cause: denylist misses admin-read Scope parsing bug Config-dependent

When Is It Exploitable?

Orders auth enabled
Public register enabled
Admin API enabled

Attack Flow

1

Unauthenticated caller registers

The attacker calls POST /api/v1/auth/register. This route is public by design when self-registration is enabled.

2

Caller asks for admin-read

The request includes scopes: ["admin-read"]. The shared scope parser recognizes it as a valid scope.

3

Parser only blocks admin-all

parse_public_scopes rejects admin-all, but does not reject admin-read.

4

JWT unlocks admin read routes

The token is accepted by routes requiring AuthScope::AdminRead, such as config, balances, whitelist, gas, fees, and rebalance reads.

Request Shape

This is the shape of the bad request. It only works when public registration is enabled.

POST /api/v1/auth/register
Content-Type: application/json

{
  "client_id": "public-client-123",
  "client_name": "Public Client",
  "scopes": ["admin-read"]
}

Why It Works

  • AuthScope includes both admin-read and admin-all.
  • parse_scopes accepts any valid AuthScope string.
  • parse_public_scopes only rejects AdminAll.
  • Admin read routes require AdminRead, so the issued token is enough.

Scope Policy: Current vs Safe

Scope Current public registration Safe public registration
read-orders Allowed Allowed if public clients need order lookup
create-orders Allowed Allowed if public clients need order submission
create-quotes Allowed Allowed if public clients need quote creation
read-quotes Allowed Allowed if public clients need quote lookup
admin-read Allowed by mistake Always rejected
admin-all Rejected Always rejected

How to Avoid This Vulnerability

Use a positive allowlist

Public registration should explicitly allow only non-admin scopes. Do not parse all scopes and then block a few dangerous ones.

Separate admin issuance

Admin scopes should only be minted after SIWE/admin whitelist verification, never from public self-registration.

Test forbidden scopes

Add tests that admin-read and admin-all both fail through /auth/register.

Recommended Code Policy

Conceptually, the public parser should be shaped like this:

fn parse_public_scopes(scopes: Option<Vec<String>>) -> Result<Vec<AuthScope>, String> {
    let scopes = parse_scopes(scopes)?;
    let allowed = [
        AuthScope::ReadOrders,
        AuthScope::CreateOrders,
        AuthScope::ReadQuotes,
        AuthScope::CreateQuotes,
    ];

    if scopes.iter().any(|scope| !allowed.contains(scope)) {
        return Err("scope is not allowed on public registration".to_string());
    }

    Ok(scopes)
}

This avoids future mistakes if new admin scopes are added later. New privileged scopes will be rejected by default until someone intentionally adds them to the public allowlist.

Operational Safeguards

  • Keep AUTH_PUBLIC_REGISTER_ENABLED=false unless public self-registration is explicitly needed.
  • If admin routes are enabled, review every path that can mint JWT scopes.
  • Monitor registration logs for unexpected privileged scope requests.
  • Consider separate JWT issuers or token audiences for public clients and admin sessions.
